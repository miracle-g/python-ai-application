# AI アプリ セキュリティチェック

Streamlit + LLM の個人/社内ツールに典型的な脆弱性を、体系的に洗い出して**深刻度順のレポート**に
まとめるためのスキル。読者は多くの場合コードを書いた本人なので、レポートは「何が・なぜ危険か・
どう直すか」が一目で分かることを最優先にする。

**このスキルは診断だけを行い、コードは書き換えない。** セキュリティ修正は影響範囲を人間が確認して
から入れるべきなので、発見と修正案の提示までに徹する（ユーザーが明示的に修正を頼んだ場合は別）。

## なぜこの構成に特化するのか

Streamlit + LLM アプリは「まず動かす」ことを優先して作られがちで、その前提が崩れる瞬間
（GitHubに上げる、社内に共有する、URLを人に渡す、外部ユーザーに開放する）に事故が起きる。
だから点検の軸は **「秘密情報が外に出ないか」と「他人の入力・LLMの出力を信用しすぎていないか」**
の2つを最優先にする。一般的なWebアプリ監査の網羅性より、この構成で現実に起きるリスクを優先する。

**アプリの規模を最初に見極める。** DBも認証もない個人ツールと、ログイン・データベース・外部公開を
持つアプリでは、必要な点検が変わる。前者なら下記7カテゴリで十分。後者なら加えて
**認証・セッション管理・SQL/NoSQL インジェクション・アクセス制御**も見る必要がある
（`references/checklist.md` の末尾に「認証・DBを持つ場合の追加観点」がある。該当する構成のときだけ読む）。
規模に対して過不足のない点検にすること——小さなツールに重装備を求めるのも、公開アプリで認証を
見落とすのも、どちらも失敗。

## 進め方

### 1. アプリの地形を把握する

まず `scripts/scan.sh` を実行して機械的な当たりを取る。これは秘密情報パターン・危険な関数呼び出し・
`.gitignore` の状態・依存関係を grep ベースで洗い出す**下調べ**であり、判断はしない。

```bash
bash <skill-dir>/scripts/scan.sh <対象アプリのルート>
```

スキャン結果は「見るべき場所のリスト」として使う。ヒット0でも安全の証明にはならない
（パターンに載っていない問題があるため）。次に自分の目で以下を確認する:

- 秘密情報の入口と出口（`.env` / `st.secrets` / `os.getenv` / `session_state` / ハードコード）
- ユーザー入力とLLM出力が**どこに描画・保存・実行されるか**（`st.markdown`, `unsafe_allow_html`, ファイル書き込み, `download_button` のファイル名）
- 外部に出る経路（gitにコミットされるファイル、公開バインド、ログ出力）
- 依存パッケージとそのバージョン

### 2. カテゴリ別に点検する

`references/checklist.md` に7カテゴリの詳細な点検項目・grep例・判断基準・修正パターンをまとめてある。
**必ずこのファイルを読んでから点検すること。** カテゴリは次の通り:

1. 秘密情報・認証情報（APIキー） — 最優先
2. 出力レンダリングと XSS（`unsafe_allow_html`）
3. 入力とインジェクション（パストラバーサル / eval / コマンド実行 / プロンプトインジェクション）
4. ファイル・データの取り扱い（履歴への機微情報蓄積、書き込み先の固定）
5. 依存パッケージの脆弱性
6. Streamlit のデプロイ・設定（XSRF / CORS / 公開バインド）
7. エラー処理と情報漏洩

各項目について「該当する / しない / このアプリでは非該当（理由）」を判断する。**「個人利用だから」で
片付けず、前提が崩れたとき何が起きるかまで一度は考える**（例: 履歴JSONLは平文だが、リポジトリに
上がらない限り問題にならない → だが `data/` が `.gitignore` にあるかは必ず確認する）。

### 3. 深刻度をつける

各発見に深刻度を1つ割り当てる。判断は「悪用の起きやすさ × 起きたときの被害」:

| 深刻度 | 目安 |
|--------|------|
| **Critical** | 秘密情報がすでに漏れている/漏れる寸前（キーがコミット済み、公開URLで認証なく機密機能が叩ける 等） |
| **High** | 悪用されれば重大だが、条件が要る（LLM出力を `unsafe_allow_html` で描画 → 反射型XSS、パストラバーサルで任意パス書き込み 等） |
| **Medium** | 特定状況で問題（機微データが平文で残る、既知CVEのある依存、スタックトレースがUIに漏れる 等） |
| **Low** | 深層防御・ベストプラクティス上の指摘（バージョン未固定、エラーメッセージが親切すぎる 等） |
| **Info** | 現状問題ないが、前提が変わったら要注意な点の記録 |

**過剰に煽らない。** 個人利用ツールに企業向けの重装備を求めるのは有害。ただし「今は安全でも、
共有した瞬間に Critical になる」ものは、その条件を明記して High 以上で拾う。

### 4. レポートを出す

会話にレポートを表示し、**同時に対象アプリのルートに `SECURITY_REVIEW.md` として保存する**
（後から見返せるように）。既存の同名ファイルがあれば上書き前に日付サフィックスを付けるか確認する。

レポートは必ず次の構成にする:

```markdown
# セキュリティレビュー: <アプリ名>

**実施日**: YYYY-MM-DD
**対象**: <パス / コミットハッシュがあれば>
**総評**: <2〜3文。全体のリスク水準と、最優先で直すべき1点>

## サマリー

| # | 深刻度 | 分類 | 概要 | 場所 |
|---|--------|------|------|------|
| 1 | Critical | 秘密情報 | ... | `path:line` |

## 詳細

### [Critical] 1. <一文で問題を要約>

- **場所**: `path/to/file.py:line`
- **何が問題か**: <事実。コードの該当箇所を短く引用>
- **想定シナリオ / 影響**: <どう悪用され、何が起きるか。具体的に>
- **修正方法**: <取るべき対処。コード例を1つ。ただし適用はしない>

（深刻度の高い順に全件）

## 良かった点

<すでに正しく守れている箇所を挙げる。安心材料になり、次の変更で壊さない目印になる>

## 対応の優先順位

1. <まず直すべきこと>
2. ...
```

**「良かった点」を省かない。** 何が既に守れているかを示すと、ユーザーは安心でき、かつ将来の変更で
それを壊さないよう意識できる。指摘だけの監査は使われなくなる。

## 判断で迷ったときの原則

- **事実と推測を分ける。** コードから確認できたことと、「たぶんこう使われる」という推測は
  レポート上で区別する。断定できないなら「要確認」と書く。
- **再現条件を書く。** 「XSSの可能性」ではなく「モデル出力に `<img src=x onerror=...>` が
  含まれ、それを `unsafe_allow_html=True` で描画すると発火する」まで書く。
- **このアプリに存在しない問題を書かない。** チェックリストは網羅のためのもので、埋めるための
  ものではない。該当なしなら詳細には載せず、必要なら「確認済み・問題なし」として良かった点に回す。

詳細な点検項目・grepパターン・カテゴリごとの修正例は `references/checklist.md` を参照。
