# app.py の読み方 — 49行のホーム画面を分解する

`app.py` には関数もクラスも出てきません。**ただ上から下に流れるだけのスクリプト**です。
それでも、このファイルには Streamlit というフレームワークの考え方と、
このアプリで一番踏みやすい地雷が両方詰まっています。順番に見ていきます。

---

## 大前提：このコードは毎回まるごと走る

普通の Web アプリなら「ページを開いたとき」「ボタンが押されたとき」で処理を書き分けます。
Streamlit にはその区別がありません。

> ユーザーが何か操作するたび、`app.py` は1行目から最終行まで、丸ごと実行し直される。

サイドバーのスライダーを動かしても、モデルを選び直しても、走るのは同じ49行です。
画面は「更新される」のではなく、**毎回ゼロから描き直されている**と考えてください。
だから `st.title(...)` のような命令は「タイトルを設定する」ではなく
「いま、ここにタイトルを1つ置く」という意味になります。上から順に紙に刷っていくイメージが近いです。

この前提が分かると、残りのコードはほぼ素直に読めます。

---

## 上から順に、何が起きているか

実際にファイルは上から下に実行されるので、その順番どおりに追います。

### 1. ページの基本設定を宣言する（14行目）

```python
st.set_page_config(page_title=APP_NAME, page_icon=APP_ICON, layout="wide")
```

ブラウザのタブに出る名前とアイコン、そして横幅を広く使う指定です。
**これは他のどの `st.` 命令よりも先に呼ぶ必要があります**。
後ろに回すと Streamlit がエラーを投げます。ページ全体の体裁は、何かを描き始める前にしか決められないからです。

名前とアイコンは直接書かず `config.py` から取り込んでいます。
アプリ名を変えたいときに探し回らなくて済むように、定義場所を1か所に寄せてあります。

### 2. サイドバーを描く（15行目）

```python
render_sidebar()
```

APIキー入力欄・モデル選択・創造性スライダーは、8つのページすべてに出ます。
その中身は `core/ui.py` に置いてあり、各ページはこの1行を呼ぶだけです。

注目してほしいのは、**戻り値を受け取っていない**ところです。
この関数は本来「選ばれたモデルと温度」を `Settings` として返しますが、
ホーム画面は文章を生成しないので設定値を使う場面がありません。捨てて構わない、という意思表示です。
各ツールのページでは `settings = render_sidebar()` と受け取っています。

### 3. 見出しを置く（17〜18行目）

タイトルと、その下の小さな説明文です。特別なことはしていません。

### 4. APIキーがあるか確認する（20〜26行目）

```python
if not resolve_api_key():
    st.warning("まずはAPIキーを設定してください。...", icon="🔑")
```

このファイルで唯一の条件分岐です。`resolve_api_key()` は
サイドバーの入力 → 環境変数 → `secrets.toml` の順に探し、どこにも無ければ `None` を返します。
その場合だけ黄色い警告が出ます。

ここで**止めてはいない**のがポイントです。警告を出したあとも処理は下に続き、カード一覧は普通に表示されます。
キーが無くてもツールを覗くことはでき、実際に生成しようとした時点で初めて怒られる、という作りです。

### 5. ツール一覧を持つ（30〜39行目）

8つのツールの情報を、タプルのリストとして手書きで持っています。
`(ファイルパス, アイコン, 名前, 説明)` の4点セットです。
このリストが次の章の主役なので、ここでは通過するだけにします。

### 6. カードを2列に並べる（41〜47行目）

リストを回してカードを描きます。ここは Streamlit 特有の書き方なので、後ろで詳しく見ます。

### 7. ヒントを畳んでおく（49〜58行目）

`st.expander` は折りたたみです。初回は開いて読んでほしいが、毎回目に入ると邪魔になる。
そういう情報の置き場所です。温度の目安や `[要確認]` の意味が書いてあります。

## TOOLS 配列は、静かに壊れる

`pages/` に置いたファイルは、Streamlit が**自動で**サイドバーのメニューに出してくれます。
ファイル名の先頭の数字が並び順、数字とアンダースコアを除いた部分が表示名です。何も登録は要りません。

ところが**ホーム画面のカード一覧は自動ではありません**。
30行目の `TOOLS` に手で書いた通りにしか出ません。
つまり同じ8ツールの情報が、ファイル名と `TOOLS` の**2か所**に存在しています。
ずれると2通りの壊れ方をします。

| ずれ方 | 何が起きるか |
| --- | --- |
| ページを追加して、`TOOLS` に書き忘れる | サイドバーには出るのに、ホームのカードには出ません。**エラーは一切出ません。** 動いているように見えるので、自分で気づくしかないのが厄介なところです。 |
| ページ名を変えて、`TOOLS` を直し忘れる | `st.page_link` が存在しないファイルを指し、`StreamlitPageNotFoundError` が飛びます。カード1枚が欠けるのではなく、**ホーム画面全体が落ちます**。 |

前者は気づけないまま放置され、後者は派手に落ちる。方向は逆ですが、原因は同じ「2か所に書いてある」ことです。
ツールを増やすときは `TOOLS` への追記までがワンセットだと覚えておいてください。

> **確認済み** — いまの `pages/` の中身と `TOOLS` の8件は一致しています。ずれはありません。

ちなみに、ファイル名に絵文字を入れていないのも意図的です。絵文字はサイドバーの表示名に混ざってしまうので、
アイコンは各ページの `setup_page("校正・推敲", "🔍")` の第2引数で渡す約束になっています。

## `cols[i % 2]` という書き方

カードを並べている部分です。ここだけは初見で戸惑うので、ばらして読みます。

```python
cols = st.columns(2)
for i, (path, icon, name, desc) in enumerate(TOOLS):
    with cols[i % 2]:
        with st.container(border=True):
            st.subheader(f"{icon} {name}")
            st.write(desc)
            st.page_link(path, label=f"{name}を開く", icon="➡️")
```

- **`st.columns(2)`** — 左右2つの「入れ物」を作り、リストで返します。
  ループの外で**1回だけ**呼ぶのが大事です。中で呼ぶと、カードごとに新しい2列が生まれてしまいます。
- **`for i, (path, icon, name, desc)`** — `enumerate` が番号 `i` を、
  丸括弧の部分がタプルの4要素をその場で展開しています。`tool[0]` のような添字を書かずに済みます。
- **`i % 2`** — 2で割った余りなので `0, 1, 0, 1...` と交互になります。
  偶数番目は左、奇数番目は右。これだけで `TOOLS` が何件あっても2列に折り返せます。
- **`with cols[...]`** — Streamlit の `with` は「以降の描画をここに入れる」という宣言です。
  このブロックの中で呼ばれた `st.subheader` などが、その列の中に置かれます。
- **`st.container(border=True)`** — 枠線がカードの見た目を作っています。
  カード用の部品があるわけではなく、枠付きの箱に3つ並べているだけです。

結果として、カードは**左・右・左・右**の順に埋まります。
`TOOLS` の並び順がそのまま画面の読み順になるので、リストの順番を入れ替えれば表示も入れ替わります。

## このファイルから持ち帰ること

- **毎回まるごと再実行される。** イベントハンドラを探しても無いのは、その概念自体が無いからです。
- **`set_page_config` は最初。** 描き始める前にしか決められません。
- **共通部分は関数1行で呼ぶ。** サイドバーの実体は `core/ui.py` にあります。
- **ホームのカードは自動同期しない。** ページを足したら `TOOLS` にも足す。ここだけは忘れると痛い目を見ます。
- **設定値は `config.py` に寄せる。** アプリ名もモデルIDも、直書きせず取り込んでいます。

次に読むなら `core/ui.py` の `generate_and_render()` をおすすめします。
各ページが「フォームだけ書けば済む」のはこの関数のおかげで、生成・表示・履歴保存を全部引き受けています。
`app.py` が一番おとなしいファイルで、あちらが一番働いているファイルです。
